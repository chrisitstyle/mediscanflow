# MediScanFlow Architecture

This document describes the architecture of MediScanFlow, a distributed medical image analysis platform.

The system is designed around an asynchronous workflow where the backend accepts medical scan uploads, stores metadata, publishes analysis requests, and receives results from a separate AI inference worker.

---

## Goals

The main goals of the architecture are:

- keep the REST API responsive,
- process AI inference asynchronously,
- separate business logic from inference logic,
- store large image files outside the relational database,
- make the local environment reproducible with Docker Compose,
- demonstrate production-oriented backend patterns in a portfolio project.

## Services

### `medical-platform-service`

The `medical-platform-service` is the main backend service.

Responsibilities:

- expose REST API endpoints,
- validate requests,
- manage patients,
- manage analysis records,
- upload original files to MinIO,
- generate presigned URLs,
- publish analysis request events,
- consume completed and failed analysis events,
- update analysis status and detections.

Technology:

- Java 25
- Spring Boot 4.1.0
- Spring Web MVC
- Spring Data JPA
- Spring AMQP
- Flyway
- PostgreSQL
- MinIO Java SDK

---

### `ai-inference-service`

The `ai-inference-service` is a Python worker responsible for AI inference.

Responsibilities:

- connect to RabbitMQ,
- consume analysis request events,
- download input images from MinIO,
- run YOLOv8n inference,
- generate annotated output images,
- upload result images to MinIO,
- publish completed or failed events.

Technology:

- Python
- Ultralytics YOLO
- OpenCV
- Pika
- MinIO Python SDK

---

### PostgreSQL

PostgreSQL stores structured application data:

- patients,
- analyses,
- detections,
- analysis status,
- model metadata,
- object storage keys,
- timestamps,
- error messages.

Large binary image files are not stored in PostgreSQL.

---

### RabbitMQ

RabbitMQ is used for asynchronous communication between the backend and the AI worker.

Exchange:

```text
mediscanflow.analysis
```

Queues:

```text
analysis.requested
analysis.completed
analysis.failed
```

Routing keys:

```text
analysis.requested
analysis.completed
analysis.failed
```

---

### MinIO

MinIO stores image files:

- original uploaded scans,
- annotated result images generated by the AI worker.

Object key format for original files:

```text
analyses/{analysisId}/{safeOriginalFilename}
```

Object key format for result files:

```text
analyses/{analysisId}/result.jpg
```

The backend uses two MinIO endpoints:

```text
internal endpoint: http://minio:9000
public endpoint:   http://localhost:9000
```

The internal endpoint is used for container-to-container communication.

The public endpoint is used to generate presigned URLs that work from the host machine or browser.

## Main Workflow

### 1. Patient Creation

The client creates a patient through the REST API.

```text
POST /api/patients
```

The backend validates the request and stores the patient in PostgreSQL.

---

### 2. Scan Upload

The client uploads an image for a patient.

```text
POST /api/patients/{patientId}/analyses
```

The request uses:

```text
multipart/form-data
```

Required part:

```text
file
```

The backend:

1. validates the uploaded file,
2. checks that the patient exists,
3. creates an analysis record,
4. uploads the original file to MinIO,
5. publishes an analysis request event,
6. returns an analysis response with status `QUEUED`.

---

### 3. Analysis Request Event

The backend publishes an `AnalysisRequestedEvent` to RabbitMQ.

The event contains:

```text
analysisId
patientId
objectKey
originalFileName
contentType
fileSizeBytes
modelName
modelVersion
requestedAt
```

The event is published after the database transaction commits.

Current implementation uses Spring transaction synchronization as a pragmatic step before introducing a full Transactional Outbox pattern.

---

### 4. AI Inference

The Python worker consumes the request event.

It then:

1. downloads the original image from MinIO,
2. loads the YOLO model,
3. runs inference,
4. extracts detections,
5. generates an annotated result image,
6. uploads the result image to MinIO.

---

### 5. Completed Event

If inference succeeds, the worker publishes an `AnalysisCompletedEvent`.

The event contains:

```text
analysisId
modelName
modelVersion
resultObjectKey
detections
completedAt
```

Each detection contains:

```text
label
confidence
x
y
width
height
```

The backend consumes this event and updates the analysis:

```text
status = COMPLETED
resultObjectKey = analyses/{analysisId}/result.jpg
detections = event detections
completedAt = event completedAt
errorMessage = null
```

---

### 6. Failed Event

If inference fails, the worker publishes an `AnalysisFailedEvent`.

The event contains:

```text
analysisId
errorMessage
failedAt
```

The backend consumes this event and updates the analysis:

```text
status = FAILED
errorMessage = event errorMessage
completedAt = event failedAt
detections = empty
resultObjectKey = null
```

## Analysis State Model

```text
UPLOADED
   |
   v
QUEUED
   |
   +----------+
   |          |
   v          v
COMPLETED   FAILED
```

Current behavior:

- new analyses are stored as queued after upload,
- the worker processes queued analyses asynchronously,
- successful inference moves the analysis to `COMPLETED`,
- failed inference moves the analysis to `FAILED`.

`PROCESSING` is reserved for a future improvement where the worker can publish an additional event when it starts processing.

objectKey

A detection represents one bounding box detected by the model.

Important fields:

## Error Handling

The backend uses a global exception handler and a consistent error response format.

Example:

```json
{
  "timestamp": "2026-06-30T16:20:00Z",
  "status": 404,
  "error": "Not Found",
  "message": "Patient not found",
  "path": "/api/patients/00000000-0000-0000-0000-000000000000",
  "validationErrors": {}
}
```

Handled cases include:

- resource not found,
- duplicate resources,
- request validation errors,
- malformed JSON,
- invalid file upload,
- missing multipart file,
- file too large,
- unexpected internal server errors.

---

## File Upload Validation

File uploads are validated before analysis records are processed.

Validation rules:

```text
file must be present
file must not be empty
file must have an original filename
filename must not contain path traversal patterns
content type must be allowed
file size must not exceed configured limit
```

Allowed content types:

```text
image/jpeg
image/png
```

Default max file size:

```text
10 MB
```

## Docker Compose Architecture

Docker Compose starts the following containers:

```text
mediscanflow-postgres
mediscanflow-rabbitmq
mediscanflow-minio
mediscanflow-medical-platform-service
mediscanflow-ai-inference-service
```

Infrastructure containers expose ports to the host:

| Container           | Host Port |
| ------------------- | --------- |
| PostgreSQL          | `5432`    |
| RabbitMQ AMQP       | `5672`    |
| RabbitMQ Management | `15672`   |
| MinIO API           | `9000`    |
| MinIO Console       | `9001`    |
| Backend API         | `8080`    |

Healthchecks are used to reduce startup race conditions between services.

## Design Decisions

### Asynchronous Processing

AI inference can be slow and resource-intensive.

The backend does not run inference directly. Instead, it publishes an event and returns quickly.

Benefits:

- REST API remains responsive,
- worker can be scaled independently,
- failed inference does not crash the backend,
- queue provides buffering between services.

---

### Object Storage for Images

Images are stored in MinIO instead of PostgreSQL.

Benefits:

- database remains focused on structured metadata,
- large binary files are handled by object storage,
- presigned URLs provide temporary controlled access,
- storage can be replaced with S3-compatible cloud storage later.

---

### Separate AI Worker

The AI worker is implemented as a separate Python service.

Benefits:

- Python ML ecosystem can be used freely,
- backend remains focused on business logic,
- model dependencies do not affect the Java service,
- service can be scaled or deployed separately.

---

### Package by Feature

The backend avoids global layer packages such as:

```text
controller/
service/
repository/
```

Instead, it groups related files by feature:

```text
patients/
analyses/
storage/
messaging/
common/
```

This keeps the codebase easier to navigate as the project grows.

## Current Limitations

The current version is intentionally focused on the core workflow.

Known limitations:

- no authentication,
- no authorization,
- no frontend,
- no OpenAPI documentation,
- no integration tests,
- no transactional outbox,
- no DICOM support,
- no user or organization model,
- no audit log,

## Summary

MediScanFlow follows a distributed architecture where the backend, message broker, object storage, database, and AI worker are clearly separated.

The core workflow is:

```text
Upload scan
→ Store original image
→ Publish analysis request
→ Run YOLO inference
→ Store annotated result
→ Publish result event
→ Update analysis status
→ Return result through REST API
```

This design keeps the backend responsive, makes AI inference asynchronous, and provides a strong foundation for future frontend, testing, authentication, and deployment work.
